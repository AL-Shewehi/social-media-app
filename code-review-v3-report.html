#C1 — router.refresh() on every realtime event kills performance

src/features/feed/components/RealtimeFeedList.tsx:47-50,57,62

Critical

🔴 The Problem & Why It's Bad

INSERT, UPDATE, and DELETE on posts, likes, and comments all call router.refresh(), triggering a full server roundtrip: the entire feed SQL query re-executes, all PostCard components re-render, and React hydration runs. This defeats the purpose of realtime — the user sees a full-page flash on every single like or comment.

💡 Proposed Solution

Use the realtime payload data for local state updates, and only call router.refresh() as a background fallback.

🛠️ Code Example

// RealtimeFeedList.tsx — optimistic local state, no router.refresh() storm
const handleRealtimePostChange = useCallback(async (payload: RealtimePostPayload) => {
  if (payload.eventType === "INSERT") {
    const newPostUserId = (payload.new as any)?.user_id;
    if (allowedUserIds && newPostUserId && !allowedUserIds.includes(newPostUserId)) return;
    // Append the new post from payload instead of full refresh
    setPosts((prev) => [{ ...(payload.new as Post), profiles: null, comments: [], likes: [] }, ...prev]);
  } else if (payload.eventType === "DELETE") {
    setPosts((prev) => prev.filter((p) => p.id !== payload.old.id));
  } else {
    setPosts((prev) => prev.map((p) => (p.id === payload.new.id ? { ...p, ...(payload.new as Post) } : p)));
  }
}, [allowedUserIds]);

📏 Principle Applied

Performance — unnecessary re-render cascade

#C2 — Optimistic UI missing — like/comment don't update until refresh

src/features/feed/components/PostCard.tsx:56-69

Critical

🔴 The Problem & Why It's Bad

handleLikeToggle sends the server action, but the UI (isLikedByMe, likesCount) only updates after revalidatePath completes. Because RealtimeFeedList calls router.refresh() on likes changes, there's a race condition: the server action revalidates, then the realtime event fires refresh again. The user sees a delayed, jumpy UI.

💡 Proposed Solution

Optimistically toggle the like state locally, then reconcile on server response with rollback.

🛠️ Code Example

const handleLikeToggle = async () => {
  if (isLikePending) return;
  setIsLikePending(true);

  // Optimistic toggle
  setLocalLiked(!isLikedByMe);
  setLocalLikesCount((prev) => (isLikedByMe ? prev - 1 : prev + 1));

  const result = await toggleLikeAction(post.id);
  setIsLikePending(false);

  if (!result.success) {
    // Rollback
    setLocalLiked(isLikedByMe);
    setLocalLikesCount(post.likes?.length ?? 0);
    toast.error(result.error || "فشل تعديل الإعجاب");
  }
};

UX — optimistic UI (missing user feedback loop)

#C3 — Middleware doesn't protect /profile/[id], /friends, and other routes

src/middleware.ts:52-54

Critical

🔴 The Problem & Why It's Bad

The middleware matcher only covers "/", "/login", "/register", "/reset-password". Routes like /profile/[id], /friends, /groups, /watch, /saved, etc. rely on individual redirect("/login") calls inside each page. This is inconsistent — if a developer forgets the check in a new page, it's exposed to unauthenticated users.

💡 Proposed Solution

Use a prefix-based pattern matcher that covers all protected routes, then redirect !user inside middleware.

🛠️ Code Example

// middleware.ts — protect all routes except public ones
export const config = {
  matcher: ["/((?!_next|api|assets|login|register).*)"],
};

// Inside middleware function:
if (!user && request.nextUrl.pathname !== "/login" && request.nextUrl.pathname !== "/register") {
  return NextResponse.redirect(new URL("/login", request.url));
}

📏 Principle Applied

Security — defense in depth (centralized vs. distributed auth)

#H1 — Every Server Action duplicates try-catch-return (DRY)

auth/actions.ts · feed/actions.ts · friends/actions.ts · profile/actions.ts · search/actions.ts

High

🔴 The Problem & Why It's Bad

Every single Server Action follows the same 7-line boilerplate: try { do work; return { success: true, data } } catch (error: unknown) { console.error("..."); return { success: false, error: error instanceof Error ? error.message : "..." } }. That's 13 actions × 7 lines = 91 lines of duplicated boilerplate. Adding a new action means copying this block again.

💡 Proposed Solution

Extract a withErrorHandling wrapper function.

🛠️ Code Example

// lib/with-error-handling.ts
"use server";

export async function withErrorHandling<T>(
  fn: () => Promise<T>,
  errorMessage: string = "حدث خطأ غير متوقع"
): Promise<{ success: true; data: T } | { success: false; error: string }> {
  try {
    const data = await fn();
    return { success: true, data };
  } catch (error: unknown) {
    console.error(errorMessage, error);
    return { success: false, error: error instanceof Error ? error.message : errorMessage };
  }
}

// Usage in any action:
export async function createPostAction(content: string, imageUrl?: string | null) {
  return withErrorHandling(async () => {
    const { supabase, user } = await requireUser();
    const { data, error } = await supabase.from("posts").insert({ ... }).select().single();
    if (error) throw error;
    revalidatePath("/");
    return data;
  }, "حدث خطأ أثناء نشر المنشور");
}

📏 Principle Applied

DRY + SRP — error handling extracted to single responsibility

#H3 — Empty string ID in FriendshipButton optimistic state causes hidden queries

src/features/friends/components/FriendshipButton.tsx:57

High

🔴 The Problem & Why It's Bad

After sending a friend request, the optimistic state sets id: "". If the user immediately clicks "إلغاء الطلب", cancelOrRemoveFriendshipAction("", targetUserId) executes: DELETE FROM friendships WHERE id = '' .... This DELETE matches nothing, silently fails, and the UI state is corrupted (shows no-relationship but server still has pending).

💡 Proposed Solution

Use null instead of "" and gate all actions on friendship?.id being truthy.

🛠️ Code Example

// Before:
setFriendship({ id: "", ..., status: "pending" });

// After — don't set optimistic state until server confirms:
setFriendship(null);

// OR use a unique optimistic sentinel:
setFriendship({ ...friendship, id: "__optimistic__", status: "pending" });
// And guard all actions:
if (!friendship?.id || friendship.id === "__optimistic__") return;

📏 Principle Applied

Defensive programming — don't lie to the type system with empty strings
